Overview

What the platform does, who uses it, and the shape of the codebase.

FarmConnect AI cuts the middleman out of local produce trade. A farmer lists what's growing; a nearby apartment community or restaurant orders it directly; a delivery partner picks it up and drops it off with an OTP handshake; an admin keeps the whole marketplace healthy. Four different people, four different jobs, one shared backend.

Farmer

Lists produce, manages incoming orders, reads AI-backed demand/price/crop guidance, checks disease photos and weather risk.

Buyer

Browses verified farms, orders by the kilogram, tracks delivery with a live OTP, rates the farm afterward.

Delivery partner

Claims ready pickups, confirms hand-off with the buyer's OTP, earns a distance-based fee automatically.

Admin

Verifies farms, moderates listings, watches platform-wide revenue and order health.

The frontend never calls the AI service directly — every AI request is proxied through the Spring backend, which persists an audit row for it. This keeps auth, rate-limiting and logging centralised in one place instead of duplicated per client.

Request lifecycle

Controller→ validates input→ Service→ business rules + @Transactional→ Repository→ Postgres

Controllers hold no business logic — they validate the request body with Bean Validation and hand off. Services own the transaction boundary and map entities straight to response DTOs inline (no separate mapper layer; the DTOs are simple enough that a mapper would add indirection without saving real code).

Data model

15 tables, all created by Flyway — every migration is forward-only and never edited once applied.

The full relational model — including tables no endpoint used yet — was built in the very first phase, specifically so later phases (Buyer, Delivery, Admin) needed only new controllers and services against tables that already existed with realistic seed data.

TablePurpose
users / roles / user_rolesOne user can hold multiple roles; local + Google auth in one table via auth_provider
farmersFarm profile, address, geocoded lat/lng, verification status
buyersApartment community or restaurant profile, delivery address, geocoded lat/lng
categories / productsProduce catalog, priced per unit, organic flag, stock status
orders / order_itemsOrder header + line items, snapshots buyer's coordinates at checkout time
deliveriesAuto-created on pickup-ready; carries the OTP and the computed delivery fee
paymentsCOD only — created/completed automatically on OTP-verified drop-off
reviews / favoritesBuyer feedback on delivered orders; saved products
notificationsIn-app feed — every entry also triggers an async email (see §07)
ai_predictionsAudit row for every AI-service call, kept for later model evaluation

Migration discipline

A migration, once applied to a running database, is never edited — Flyway checksums the applied file and refuses to start if it's changed underneath it. Every correction during this build shipped as a new numbered migration instead:

03

Auth & access control

Stateless JWT for everyone, Google OAuth2 layered on top — and only switched on once real credentials exist.

Local login

BCrypt-hashed passwords. POST /api/auth/login returns an HMAC-SHA256 JWT (24h expiry) carrying userId and roles. Every protected request carries it as Authorization: Bearer <jwt>; JwtAuthFilter runs before Spring's own UsernamePasswordAuthenticationFilter and populates the security context directly from the token — there's no server-side session at all.

Google OAuth2 — wired up, but conditionally

Spring Boot's own OAuth2ClientAutoConfiguration refuses to even start the app if a client registration exists with a blank client-id. So the Google client isn't declared under Spring's managed spring.security.oauth2.client.* path at all — it's read as a plain custom property (app.oauth2.google.*) and the ClientRegistration is built by hand inside SecurityConfig, and .oauth2Login() is only attached to the filter chain when both GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET are non-blank:

SecurityConfig.java
if (StringUtils.hasText(googleClientId) && StringUtils.hasText(googleClientSecret)) {
    http.oauth2Login(oauth2 -> oauth2
            .clientRegistrationRepository(googleClientRegistrationRepository())
            .successHandler(oAuth2LoginSuccessHandler));
}

Until both values exist, /oauth2/** simply 404s and "Continue with Google" is inert — the rest of the app works normally on email/password. This same wired-up-but-inert shape is reused for every optional integration in §07: Cloudinary, SMTP, and Google Maps all boot cleanly with blank credentials and quietly activate the moment real ones appear.

New Google sign-ins with no role yet

A brand-new Google account gets a short-lived (15 min) "onboarding" JWT and is routed to a role-selection screen, which calls POST /api/auth/select-role to trade it for a full access token.

Authorization

@PreAuthorize("hasRole('FARMER')") on controllers, matching ROLE_* authorities minted directly from the roles table — no separate mapping layer. A request with a valid JWT but the wrong role gets a clean 403; a request with no JWT at all gets a clean 401 — see the entry-point bug in §08 for why that second guarantee needed an explicit fix.

The four dashboards

One shared shell — Navbar, Sidebar, ProtectedRoute — four different sidebars and route trees layered on top.

Every dashboard shares the same Navbar component (it takes notificationsPath / profilePath / cartPath props so it can point at the right role's routes) and the same ProtectedRoute gate, which checks the JWT's roles client-side before ever rendering the page. Log in as any role and resolveDashboardPath(roles) sends you to the right home screen automatically.

Delivery/delivery/*Dashboard, Available Pickups, My Deliveries, History, Earnings, Profile

Buyer cart

The cart is deliberately client-side only, persisted to localStorage via CartContext, and scoped to a single farmer at a time — adding a product from a second farm clears the first farm's items, matching how a single order can only belong to one farmer (orders.farmer_id is a single foreign key, not a per-line-item one). There's no separate cart API; checkout sends the whole cart as one POST /api/buyer/orders call.

05

Order & delivery lifecycle

A five-state order machine with an OTP hand-off baked into the middle of it.

stateDiagram-v2
  [*] --> PENDING : buyer places order
  PENDING --> ACCEPTED : farmer accepts
  PENDING --> REJECTED : farmer rejects
  ACCEPTED --> READY_FOR_PICKUP : farmer marks ready\n(Delivery row + OTP generated here)
  READY_FOR_PICKUP --> OUT_FOR_DELIVERY : partner claims + picks up
  OUT_FOR_DELIVERY --> DELIVERED : OTP verified at drop-off
  REJECTED --> [*]
  DELIVERED --> [*]

Where the OTP comes from

There is no manual dispatch step. The instant a farmer marks an order READY_FOR_PICKUP, the backend inline-creates the Delivery row: a random 4-digit OTP, and a delivery fee computed from a Haversine distance between the farmer's and buyer's geocoded coordinates.

delivery fee formula
fee = ₹30 base + ₹5 × distanceKm(farmer.lat/lng, buyer.lat/lng)

Any delivery partner can then claim it from /delivery/available. The OTP is surfaced to the buyer on their own order-detail page — but only while the order is in READY_FOR_PICKUP or OUT_FOR_DELIVERY — and is the one thing the delivery partner needs from the buyer at the door to call PATCH /api/delivery/{id}/complete, which marks the order DELIVERED and the payment COMPLETED in a single transaction.

Why Haversine, not a Maps distance API

Delivery fee and "nearby farmers" filtering both need a distance between two known coordinate pairs — that's pure spherical geometry, not a routing problem, so it's computed in-process with the Haversine formula rather than spending a paid API call on it. Google's Geocoding API is reserved for the one thing Haversine can't do: turning a typed address into coordinates in the first place (§07).

06

AI features

Five features, each behind a swappable interface — today's implementations are honest heuristics, not fake ML dressed up.

The frontend never calls the AI service directly; every call is proxied through /api/ai/** on the Spring backend, which persists an ai_predictions audit row and can apply auth centrally. Inside ai-service/app/ml/, each feature sits behind an abstract base class — swapping in a real trained model later means implementing that interface and changing one line, not touching the router or the Java side at all.

FeatureCurrent implementation
Demand predictionWeighted moving average over recent order history × a month-based seasonality multiplier
Crop recommendationLookup table keyed by (soil type, season), ranked by static suitability score
Price predictionCategory base price × organic multiplier × seasonality multiplier − bulk-quantity discount
Disease detectionDeterministic hash of the uploaded image buckets into one of 9 seeded conditions — same photo always returns the same result. The explicit swap point for a real CNN.
Weather alertsLive — see §07. Falls back to a deterministic synthetic provider if the API key is unset or the call fails.
07

Real integrations

Four third-party services layered on in one working session — email, weather, maps, image hosting. Each follows the same rule: wired up fully, inert until real credentials exist.

Email — Gmail SMTP + async delivery

Nine real events already wrote to the notifications table before this — order placed, accepted, rejected, ready-for-pickup, claimed, picked up, delivered, paid, reviewed. Rather than scatter email calls across all seven call sites, the hook sits in one place: NotificationServiceImpl.notify() saves the in-app row and then fires the email — every existing trigger gets an email for free.

NotificationServiceImpl.java
notificationRepository.save(notification);
emailService.sendNotificationEmail(user.getEmail(), user.getFullName(), title, message, type);

Sending runs on a dedicated @Async("emailExecutor") thread pool (2–4 threads, AsyncConfig.java) so a slow SMTP handshake never blocks the request that triggered it. The message body is one shared branded HTML template (EmailTemplateBuilder) — a green header with the FarmConnect wordmark, a colored badge per notification type (order / payment / AI / weather), and a "thanks for growing with FarmConnect AI" footer — built as a Java text block with inline table-based styling so it survives Gmail and Outlook stripping <style> blocks.

MAIL_PORT587 (STARTTLS)

Verified live: accepted a real test order and watched the branded email land in a real inbox within seconds.

Weather — OpenWeatherMap

OpenWeatherMapProvider calls OpenWeatherMap's free Current Weather endpoint for real temperature, humidity and condition. That free tier has no alerts feed (that's the paid One Call API), so alerts are derived from the real numbers instead of being invented: temperature ≥ 40°C raises a heat-wave alert, "thunderstorm" in the condition raises a storm alert, "rain"/"drizzle" raises a heavy-rain alert, humidity ≥ 85% raises a fungal-risk alert. Any network failure falls straight back to the original synthetic provider — the farmer always gets an answer.

Verified live against Pune coordinates: real city name, real temperature, real cloud condition.

Maps — Geocoding + live map display

Farmer and buyer profiles used to require typing raw decimal latitude/longitude by hand. Now saving a profile geocodes the typed address server-side through GeocodingClient (Google's Geocoding API) and overwrites the coordinates automatically — the manual number inputs are gone from both profile forms, replaced by a live <LocationMap> component (built on @react-google-maps/api) showing exactly where the address resolved to.

FarmerServiceImpl.updateProfile()
Optional<Coordinates> geocoded = geocodingClient.geocode(buildAddress(address, city, state, pincode));
if (geocoded.isPresent()) {
    farmer.setFarmLatitude(geocoded.get().latitude());
    farmer.setFarmLongitude(geocoded.get().longitude());
} else {
    // Maps not configured, or lookup failed — keep whatever the client sent as a fallback
    farmer.setFarmLatitude(request.farmLatitude());
    farmer.setFarmLongitude(request.farmLongitude());
}

The buyer's Nearby Farmers page also gained an actual map above the card grid, pinning every farmer that has resolved coordinates — the same distance math from §05 now has real coordinates behind it instead of hand-typed ones.

Verified live: entered "Shaniwar Wada, Pune" as a farm address and the resulting pin landed exactly on the correct real-world location.

Cloudinary — image hosting

Product and profile photo uploads already had a fallback path to local disk (backend/uploads/) for when Cloudinary isn't configured. Wiring in real credentials exposed a real gap in that fallback — see §08.

08

Bugs found & fixed

Every real defect found while wiring these integrations into a live, running system — not a curated highlight reel.

Enabling Google login silently broke API session-expiry handling

auth flowSpring config

The moment .oauth2Login() was switched on, Spring Security's default response for any unauthenticated request — not just a browser page load — became a 302 redirect to Google's consent screen instead of a plain 401. The SPA's Axios interceptor expects a JSON 401 to redirect to /login on an expired token; a silent redirect to accounts.google.com would have broken that entirely.

Fix: an explicit exceptionHandling().authenticationEntryPoint(...) in SecurityConfig that forces a bare 401 for the whole filter chain. Direct navigation to /oauth2/authorization/google — the actual "Continue with Google" button — still works, since that's a request to a permitted endpoint, not something bounced through the entry point.

Async email sending crashed with a Hibernate thread-safety violation

correctnessHibernate

The first working version passed the live JPA User entity into the @Async email method. Since @Async runs on a different thread than the one holding that entity's Hibernate session, touching user.getEmail() from the async thread raced the original transaction and threw IllegalStateException: Illegal pop() with non-matching JdbcValuesSourceProcessingState — which also corrupted the original request's transaction, rolling back the order-accept it was part of.

Fix: extract email/fullName as plain strings on the calling thread before invoking the async method, so the async thread only ever touches immutable Java objects, never a Hibernate-managed entity.

Cloudinary permission error crashed product creation instead of falling back

resilienceimage upload

Cloudinary rejected the real API key with Request forbidden due to missing permissions — a scoped-key/account issue on Cloudinary's side. The existing fallback-to-local-disk logic only caught IOException; Cloudinary's SDK actually throws a plain RuntimeException for API-level failures, so it propagated straight through as a 500 instead of triggering the fallback it was written for.

Fix: broadened the catch clause to Exception. Product creation now degrades to local-disk storage on any Cloudinary failure — network, auth, quota, or permissions — exactly as originally intended.

Two broken seeded product images

dataseed data

"Okra (Bhindi)" and "Fresh Coriander" both shipped with Unsplash photo IDs that now 404. Found via a scripted UI sweep across every dashboard, not by eyeballing screens.

Fix: V24 — a corrective migration updating just those two image_url values to verified, currently-live Unsplash photos.

Two backend instances answering on different ports

infralocal dev

An older backend process, started earlier without the new Google OAuth env vars, was still running on port 8081 — and the frontend's .env pointed there, not at the freshly-restarted instance on 8080 that actually had the credentials. "Continue with Google" kept 500-ing even after the credentials were correctly set, because the process actually serving requests didn't have them.

Fix: killed both stray processes, standardized on port 8080 (matching the registered Google redirect URI), and restarted a single canonical backend + frontend pair.

.env failed to load — unquoted < in MAIL_FROM

toolingshell

MAIL_FROM=FarmConnect AI <no-reply@farmconnect.ai> is valid as a value, but source .env in bash parses the file as shell script — the unquoted < is interpreted as input redirection, breaking the whole source command before the backend even started.

Fix: quoted the value — MAIL_FROM="FarmConnect AI <no-reply@farmconnect.ai>".

A seeded demo account lost its role assignment

datalocal DB drift

farmer1@farmconnect.ai logged in successfully (valid JWT, 200 response) but the token's roles claim came back empty — every other seeded account had exactly one role row, this one had none. The seed migration itself was correct; this was local database drift on one machine, not a code bug.

Fix: restored the missing user_roles row directly.
